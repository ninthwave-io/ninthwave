# Bun consuming 43GB RAM during parallel worker orchestration

**Date:** 2026-03-25
**Severity:** critical
**Component:** worker execution / bun runtime

## Observation

During the third orchestration batch (H-REM-1, H-REM-2, M-DX-1), bun was observed consuming ~43GB RAM. This is massively beyond the expected ~1GB per worker. With 3 workers + orchestrator, total bun memory should be ~4-5GB, not 43GB.

## Context

- Machine has 24GB physical RAM (per wip_limit_resolved log). 43GB means heavy swap pressure.
- Multiple parallel workers each running `bun test` in separate worktrees.
- Workers also had hook errors (PreToolUse:Edit, PostToolUse:Edit) visible in the screen output -- unclear if related.
- Memory-aware WIP was set to 4, but that's based on the ~1GB/worker estimate. Real consumption is 10x higher.

## Hypotheses

1. **bun test memory leak in worktrees:** Each worker runs `bun test test/` which may have memory issues when run from worktrees (different `node_modules` resolution?).
2. **Multiple bun processes compounding:** Orchestrator (bun) + 3 workers (each spawning bun for tests) + Claude Code language servers = many bun processes.
3. **Test suite memory issue:** The test suite itself may have a leak -- tests import large modules or don't clean up properly.

## Investigation (performed live)

Identified: PID 31227 -- `bun test test/orchestrate-remote.test.ts` (H-REM-2 worker's test file).
- 73.8% CPU, 13% RAM resident (3.2GB), **561GB virtual memory**
- The test file was 446 lines generated by the H-REM-2 worker -- likely contains an infinite polling loop or timer that never terminates, causing unbounded memory growth.
- Killed with `kill -9 31227`. Orchestrator and status watcher processes were unaffected.

## Root cause

Worker-generated test code with a runaway loop. This is not a bun bug -- it's a worker writing bad tests. The orchestrator needs a memory/time watchdog for worker test runs.

## Suggested fixes

1. **Test timeout enforcement:** Workers should run tests with a timeout (`bun test --timeout 60000`). The orchestrator or worker prompt should mandate this.
2. **Memory watchdog:** The daemon should monitor child process memory (via `ps`) and kill any process exceeding a threshold (e.g., 4GB resident).
3. **Better test review in worker prompt:** The worker agent prompt should explicitly warn against polling loops, infinite timers, and missing cleanup in tests.

## Impact

Critical -- a single bad test consumed 561GB virtual / 3.2GB resident and caused severe swap pressure on the machine. Manually killed.
